# プロジェクト要件定義書：次世代パラメーター調整用スプレッドシート（Logic-Separated Spreadsheet）

## 1. プロジェクト概要
本プロジェクトは、データ（値）とロジック（計算式）を完全に分離した、新しいアーキテクチャのWebベース・スプレッドシートを開発する。
ゲーム開発におけるダメージ試算やバランス調整など、複雑な計算モデルを構築する際、既存のエクセルの「セル内に数式が隠蔽される」問題を解決し、AIとの協業（コードの自動生成・レビュー）やGitによるバージョン管理を容易にすることを目的とする。

## 2. 技術スタック
- **フロントエンドフレームワーク**: React (TypeScript) + Vite
- **データグリッド (UI)**: AG Grid (Community版)
- **計算エンジン**: HyperFormula (エクセル互換のHeadless計算エンジン)
- **コードエディタ**: `@monaco-editor/react` (生の関数テキストを編集・表示)
- **スタイリング**: Tailwind CSS

## 3. コア・アーキテクチャ設計（Single Source of Truth）
状態管理において、データと関数を別々のステートとして保持する。

- `dataState`: セルの値に加えて「現在そのセルに関数が入っているか」を見分けるためのフラグ（`isFormula`）等を含む構造体の2次元配列。（例: `{ value: 100, isFormula: false }`）
- `formulaState`: 生の数式が改行区切りで記述されたプレーンテキストデータ。**これが本システムにおけるロジックの唯一の情報源（Single Source of Truth）となる。** 記述の上下順序には依存せず、HyperFormulaによって自動的に関数型言語ライクに依存関係が解決される。循環参照が発生した場合はエラー（#CYCLE!）として扱う。
- `styleState`: (将来構想) CSSライクにセルの背景色や条件付き書式を定義するテキストデータ。

## 4. 関数部（ロジックエリア）の詳細仕様
関数部は単なるエクセルの数式ではなく、**「出力先セル = 計算式」**というプログラミングライクな構文で記述・保存される。

### 4.1. 基本構文（A1参照モード）
HyperFormulaに渡す前の独自構文として、行ごとに以下のように記述する。
```text
A23 = V23 + L23
B1 = SUM(A1:A10)
C2 = IF(A2 > 100, "Critical", "Normal")
```

### 4.2. エイリアス（名称定義）機能
可読性を高めるため、特定の列や単一のセルに対して名称（エイリアス）を付け、変数のように扱う機能を実装する。
システム内部（プレプロセッサ）で正規表現やAST（抽象構文木）を用いて、これをA1参照に変換してからHyperFormulaに評価させる。

**記述例:**
```text
// 変数定義（エイリアスマッピング）
ALIAS [基本攻撃力] = V
ALIAS [武器補正] = L
ALIAS [最終ダメージ] = A

// 計算ロジック
[最終ダメージ]23 = [基本攻撃力]23 + [武器補正]23
// ↑ 内部パーサーで「A23 = V23 + L23」に変換されて計算される
```

## 5. UI/UXの要件（デュアルビュー機構）
画面は「スプレッドシートビュー（表計算）」と「コードビュー（関数エディタ）」の2つのペインを持ち、ユーザーの用途に合わせて表示を切り替え（または左右分割表示）できる。

### 5.1. スプレッドシートビュー（通常モード）
見た目と操作感は既存のエクセルやGoogleスプレッドシートと完全互換とする。
- **数式バー**: 画面上部に配置。
- **セル選択時の挙動**: ユーザーがAG Grid上でセル（例: `A23`）を選択した際、システムは裏側の `formulaState`（テキスト）から `A23 = ...` または `[最終ダメージ]23 = ...` で始まる行を検索する。
- **表示**: 該当行が見つかった場合、その右辺（例: `V23 + L23` またはエイリアス表記）を数式バーに表示する。数式が存在しない場合はセルの純粋な値を表示する。
- **空の式と削除**: AG Grid上でセルの値を削除（空にする）したり、直接リテラル値で上書きした場合は、`formulaState` から該当行を削除し、`dataState` の関数フラグ（`isFormula`）を外す。

### 5.2. コードビュー（全体俯瞰モード）
関数全体を管理するためのモード。
- Monaco Editorを使用し、`formulaState` の全テキストを表示する。
- シンタックスハイライトを実装し（左辺、等号、関数名、変数名などを色分け）、可読性を高める。
- **エラーハンドリング**: 関数に循環参照などのエラーがあった場合、Monaco Editor上の該当行に赤いマーカー（波線等）を表示し、さらにAG Grid側の該当セルの背景を赤くハイライトして `#ERROR` のように警告を表示する。

## 6. 双方向同期（Two-way Sync）のメカニズム
表計算UIとテキストエディタ間の完全な同期を実現する。

1. **数式バーからの編集**:
   - ユーザーが `A23` セルを選択し、数式バーで `V23 + L23 + 10` に書き換えてEnterを押す。
   - `formulaState` の該当行のテキストが自動的に `A23 = V23 + L23 + 10` に置換（または新規追加）される。
   - 更新されたテキストをプレプロセッサが処理し、HyperFormulaが再計算を行い、AG Gridの表示が更新される。
2. **コードビューからの編集**:
   - Monaco Editor上で数式を直接書き換える、あるいは複数行をコピペ・一括置換する。
   - テキストの変更（`onChange` イベント）を検知し、即座にHyperFormulaに新しいルールセットとして流し込む。
   - AG Grid上の該当セルの値がリアルタイムに再計算され、画面が更新される。

## 7. 開発フェーズ（AIへの指示ステップ）
Vibe Codingで実装を進めるためのステップ。AIには段階的に実装を指示すること。

- **Phase 1 (Core Engine & Grid)**: AG GridとHyperFormulaの連携。`dataState`（ダミーデータ）を読み込み、画面に表を描画する最小構成を作成する。
- **Phase 2 (Basic Sync)**: Monaco Editorを配置し、プレーンなA1参照構文（`A1=B1+C1`）のパース処理と、テキストエディタ↔AG Grid間の双方向同期を実装する。
- **Phase 3 (UI Refinement)**: スプレッドシートビューの上部に数式バーを追加し、セルクリックで該当する関数テキストを検索・表示・編集するUIを実装する。
- **Phase 4 (Alias System)**: 正規表現を用いたエイリアス（名称定義）機能のプレプロセッサを実装し、日本語変数名での計算を可能にする。
